# Fudge Ur Uncle

Politician accountability app. Tracks federal + state legislators, follows campaign donations, correlates them with upcoming votes, and surfaces alerts. Mobile-first React frontend on FastAPI backend, deployed as a Vercel PWA + Procfile-based backend host.

## Repo Layout

```
backend/
  server.py          # FastAPI entrypoint, all HTTP endpoints
  config.py          # API keys + HOST/PORT from .env
  db.py              # SQLite schema, migrations, connection helper
  Procfile           # web: uvicorn server:app --host 0.0.0.0 --port ${PORT:-8000}
  data/whoboughtmyrep.sqlite   # local DB (gitignored)
  api/               # external API wrappers (one module per upstream)
    legislators.py openfec.py congress_gov.py whoboughtmyrep.py
    events.py news.py guardian.py ai_summary.py
    stance_analysis.py promises.py ai_cache.py
    legiscan.py state_sites.py followthemoney.py ftm_directory.py
    auth.py alerts_router.py upcoming_votes_router.py
  alerts/            # alert generation pipeline
    config.py industry_map.py scoring.py catcode_map.py
    ingest_fec.py ingest_ftm.py ingest_federal_votes.py ingest_state_votes.py
    pac_classifier.py reclassify.py state_categories.py
    pipeline.py refresh.py seed.py
  data/ftm_eids.csv  # curated (state, chamber, name, eid) directory
  tests/             # pytest smoke + pipeline + classifier
frontend/
  vite.config.js     # proxies /api -> localhost:8000
  vercel.json        # SPA rewrites, no-cache on /sw.js
  public/            # manifest.webmanifest, sw.js, icons
  src/
    main.jsx api.js
    App.jsx          # all 24 screens + routing in one file (deliberate)
    groupAlerts.js   # client-side alert grouping (extracted for unit tests)
    copy.js          # centralized user-facing strings (tone swap-point)
```

## Run Commands

From project root (imports require it):

```
# Backend
cd backend && pip install -r requirements.txt && python server.py     # :8000
# Frontend
cd frontend && npm install && npm run dev                              # :5173
# Pipeline (one-off)
python -m backend.db                       # init schema, first run only
python -m backend.alerts.refresh           # cron entrypoint: federal+state ingest then pipeline
# Tests
cd backend && python -m pytest tests/ -v
cd frontend && npm test                    # vitest, currently covers groupAlerts only
```

`refresh` chains `ingest_federal_votes` → `ingest_state_votes` (default `CT,NY,NJ,CA,MA`) → `pipeline`. Flags: `--states CT`, `--skip-federal`, `--skip-state`, `--congress 119`. Ingester failures log but don't block; only a pipeline crash exits non-zero.

Don't run `seed.py` and `refresh` against the same DB — `refresh`'s ingest purges scheduled-vote rows not in the live keepers list, including seeded ones. Pick one source.

Tests use `FUU_DB_PATH` (set by conftest) so they don't clobber dev data.

## Environment

`backend/.env` (all optional — missing keys fall back to sample data or DEMO_KEY):

- `DATA_GOV_API_KEY` — Congress.gov + OpenFEC (api.data.gov/signup)
- `OPENAI_API_KEY` — gpt-4o-mini for stances, promises, event summaries
- `NEWSAPI_KEY` — primary news source for event detail (100 req/day free)
- `GUARDIAN_API_KEY` — fallback news source
- `WHOBOUGHTMYREP_API_KEY` — industry-attributed funding
- `LEGISCAN_API_KEY` — state legislators (30k/mo + 1 req/sec, cached in `ai_cache`)
- `FTM_API_KEY` — FollowTheMoney/NIMP state campaign finance

`GET /` reports all 7 keys (`data_gov`, `whoboughtmyrep`, `legiscan`, `ftm`, `openai`, `newsapi`, `guardian`) — Settings reads this to render the "Keys configured" list, so keep it in sync when adding new env-dependent features.

## Architecture

**Backend is a thin aggregator.** External APIs wrapped in `backend/api/*.py`; `server.py` composes them. Only the alerts subsystem persists to SQLite — everything else is fetched live with in-process caching.

**Frontend has graceful fallback.** Every screen wraps API calls in try/catch and falls back to embedded sample data with an "OFFLINE" badge. Don't remove — it's how the app stays demoable. `apiRequest` auto-clears localStorage on 401 (only when a token was actually sent), preventing dead-token retry loops.

**Single-file frontend.** `App.jsx` holds all 24 screens. Inline `style={s.foo}` against a shared `s` map. Screen routing via `currentScreen` state + `SCREENS` enum, not React Router. Don't split casually. Two narrow exceptions are extracted modules: `groupAlerts.js` (testable in isolation, covered by vitest) and `copy.js` (centralized user-facing strings — single swap-point for tone, currently leans "friendly to new voters / immigrants" rather than the original accountability framing). `COPY` currently covers Auth (`auth.splash` / `auth.createAccount` / `auth.login` — entry-funnel headers, form labels, button idle/busy states, footer links incl. guest-mode CTA), Onboarding (`onboarding` — issue-select title, selection counter, save-button states), Dashboard, Alerts (federal + state), Events (list + detail), Settings (issues editor block), Politician Profile (score cards + nav tiles + `votingHistory` / `promiseScoring` / `timeline` sub-screens), State Rep Profile (tile labels + `voting` / `stances` / `promises` sub-screens + `recentSponsored`), Take Action (header + action buttons + call-script template), Contact-your-reps (title + call/website buttons + aria-labels), and Learn-to-Vote (state-specific guide cards + nationwide resource list). The previous all-caps "terminal" tone is gone app-wide — Events `AI summary` / `Schedule` / `Committees` / `Witnesses` / `Legislation` / `News coverage`, `alerts.urgentBadge: "Urgent"`, `events.offlineBadge: "Offline — sample data"`, the profile tile labels (`Funding breakdown` / `Voting record` / `Voting positions` / `Activity timeline` / `Contact / take action`), the sub-screen headers they navigate to (now sourced from the same `COPY` keys so tile and page header stay in sync), and the entry-funnel headers (`Create account` / `Welcome back` / `What issues matter most?` / `Done — show me my reps`) were sentence-cased in the whole-app warm pass alongside `s.sectionTitle` dropping `textTransform: uppercase`. Category filter chips inside `VotingHistoryScreen` and `StateRepVotingScreen` render via `friendlyCategory()` / `friendlyCategoryInline()` so raw keys (e.g. `foreign_policy`) display as "Foreign policy" / "foreign policy" — don't reintroduce naive `charAt(0).toUpperCase()` or raw-key fallbacks. **Typography rule:** `s.headerTitle` / `s.sectionTitle` / `s.btn` / `s.input` / `s.chip` / `s.badge` / `s.navItem` / `s.backBtn` all use `fontSans` (IBM Plex Sans); inline `fontFamily: font` (IBM Plex Mono) is reserved for data — dollar amounts, vote dates, bill IDs, score numbers, score-chip pills, chamber/AI-summary letter-spaced badges, `<code>` blocks, avatar/party glyphs, and the dev-only "Live Prototype" outer chrome. Don't re-introduce mono on form labels, subtitles, error/status messages, empty states, or attribution notes — those went sans in the typography sweep. Inline placeholders and one-off validation errors (`"Email is required"` etc.) stay hardcoded in `App.jsx` — too terse and edge-case to centralize. New user-visible strings on any covered screen should go through `COPY`.

**Learn-to-Vote uses `STATE_VOTING_GUIDE` from `copy.js`.** Keyed by 2-letter state code; covers CT/NY/NJ/CA/MA (the 5 jurisdictions the backend tracks). Each entry carries `name`, `registerUrl`, `pollingPlaceUrl`, `officialUrl`, `registrationDeadline`, `idRequired`, `pollingHours` — all user-facing strings, so swap from the same module as other tone. Screen reads `userState`, shows the state card (facts + 3 state-specific links) if present, falls back to `learnToVote.genericNote` otherwise; the nationwide `learnToVote.resources` list always renders below. Adding a new state: append an entry to `STATE_VOTING_GUIDE` and the screen picks it up. State voting rules change frequently — `learnToVote.sourceNote` surfaces a "verify on the official site" disclaimer below each state card, and the official state-site link is always one of the three CTAs so the user has a path to the source of truth.

**Dashboard is an orientation surface, not an alerts feed.** The home screen targets new voters / immigrants learning the system, not the original accountability-watchdog audience. Order is Start here (quick actions) → Coming up → Your reps. The greeting ("Hey {first}" / "Hi there") lives inside the scrollable body — not a fixed header band — so it scrolls away with content. Section titles use `s.sectionTitleFriendly` (sans 14/600, sentence case) here; the rest of the app uses `s.sectionTitle` which is also sans + sentence case (13/600) now — the whole-app warm pass retired the previous mono-uppercase-letterspaced "terminal" variant. "Coming up" calls `/api/upcoming-votes` (state-scoped, optionally category-filtered, returns vote rows with a `days_until` field) and renders the first 3 with a relative-time line. When the signed-in user has selected issues, the greeting subline folds them in ("…in CT around healthcare, housing") and the same issue keys are passed as the `categories` query param so the card is filtered to their topics. The old red URGENT banner was removed: it duplicated the Alerts-tab destination and read as scandal-detection on a learning surface.

**Warm-card surfaces bypass `s.card`.** Three dashboard components — `ComingUpCard`, `RepCardShell`, `QuickActionButton` — share a visual signature that doesn't read off the `s` map: white `colors.surface` background, 1px `colors.border`, 14–16px radius, accent-tinted low-contrast diffusion shadow (`0 6px 18px -10px rgba(231,122,27,0.13)` + a 1px highlight line). `RepCardShell` and `QuickActionButton` still use JS-tracked pointer-press feedback (`onPointerDown/Up/Leave/Cancel` → `translateY(1px)` + shadow collapse). `ComingUpCard` was refactored: the outer warm surface is now a non-interactive `<div>` and each `ComingUpRow` is its own `<button>`, so per-row press feedback comes from the global `button:active` CSS rule (keyboard + screen-reader navigation per row, no JS state). Everywhere else still uses the cream `s.card` pillow (radius bumped 10 → 14 in the whole-app pass; mono font swapped to sans on `btn`/`input`/`chip`/`badge`/`navItem`/`backBtn`/`headerTitle`). Don't mix the two on the same surface — pick one based on whether elevation is doing hierarchy work. The imminent-vote row inside `ComingUpCard` gets a soft `colors.accentDim` background tint plus a small pulsing accent dot; the previous orange left rail was removed for being too alarm-y.

**Motion conventions.** A module-level `<style>` injection near the top of `App.jsx` (id `fuu-anim-styles`, idempotent against HMR) defines three keyframes — `fuu-fade-up` (8px settle), `fuu-shimmer` (skeleton background pan), `fuu-soft-pulse` (imminent-vote dot) — plus a global `button:active, [role="button"]:active { transform: translateY(1px) }` press-feedback rule and a `@media (prefers-reduced-motion: reduce)` blanket that flattens animation/transition durations to 0.01ms. `s.body` carries `animation: fuu-fade-up 0.32s cubic-bezier(0.16, 1, 0.3, 1) both` so every screen fades up on mount. For staggered list reveals on data arrival, wrap each row in `<FadeIn delay={Math.min(i * N, cap)} duration={...}>` — `FadeIn`'s default `duration` is 300ms (the per-item swiftness knob); `delay`'s `N` is the per-item gap (the cascade-speed knob). Current callsites: Dashboard reps `i*130`/cap 780 (deliberately wider than other surfaces because 5–7 cards otherwise compress into a pop), Search results `i*55`/cap 495, Alerts groups `i*70`/cap 630, Events list `i*70`/cap 630, StateReps `renderCard` `i*55`/cap 495. `ComingUpRow` runs its own internal stagger (`240ms` duration, `80 + index * 95ms` delay) since it lives inside a parent that fades in on its own data-arrival cycle. Profile screen also cascades in two stages: 3 score cards at 0/90/180ms, then 5 nav tiles starting at 240ms with 110ms gaps. **Tuning intent:** items snap in fast, cascades read as deliberate waves — bumping per-item duration over ~350ms makes the page feel sluggish; tightening the cascade gap below ~55ms collapses the wave into a pop. **Performance gotcha:** `RepCardShell` and `QuickActionButton` track press state in React (`useState` + pointer handlers) — fine because there's no perpetual motion in them. `ComingUpCard` no longer does (rows are buttons relying on global CSS press feedback). Don't ladder more JS-driven motion onto these surfaces; if you need infinite/continuous animation, use a fresh CSS keyframe in `fuu-anim-styles` and let the GPU handle it.

**`users.issues` storage standardized on backend category keys.** `IssueSelectScreen` (onboarding) and the new Settings issues block both render labels via `friendlyCategory()` / `COPY.categories` but persist the 14 lowercase keys (`healthcare`, `environment`, `foreign_policy`, etc.) — the same set `scheduled_votes.category` uses. This lets `/api/upcoming-votes?categories=...` accept the stored list directly without translation. Legacy display-string values ("Healthcare", "Workers' Rights") from earlier builds are tolerated on read: the upcoming-votes endpoint intersects against `VALID_CATEGORIES` and ignores unknowns, and the frontend filters them out before rendering chips. No migration script — only one local user existed. Post-signout/delete defaults are also lowercase keys. Fresh signups start with `[]` (no pre-selection) and the Done button is disabled until at least one chip is picked — the prior default of `["healthcare", "environment"]` silently persisted categories the user never chose.

**`/api/upcoming-votes` is the dashboard's read path.** Reads `scheduled_votes` directly: `state` query param (federal rows always included; state side narrowed), `categories` (comma-separated, validated against the 14-key allowlist, 400 on unknown), `limit` (≤100). Optional bearer auth via `get_current_user_optional` (auth.py): when present, the endpoint defaults `categories` to the user's stored `issues` ∩ `VALID_CATEGORIES`; explicit `?categories=` overrides. Response shape: `{count, filters: {state, categories, personalized}, votes: [{id, jurisdiction, state_code, bill_number, title, category, scheduled_date, chamber, days_until}]}`. The `personalized` flag lets the frontend show why the feed is narrowed.

**Alerts are grouped client-side.** Pipeline writes one row per (donation × vote) pair, so a single donor in one category produces N near-identical cards (same headline, only bill differs). `groupAlerts()` (extracted to `frontend/src/groupAlerts.js` so it can be tested in isolation; covered by `groupAlerts.test.js` under vitest) collapses by `(actor_id, industry, category)`, takes the highest-scoring row as the lead, and rolls the rest into a bill list ("N upcoming {category} bills · $X lifetime" + first 3 bills + "(+K more)"). Both `AlertsScreen` and `StateRepAlertsScreen` use it. The inline `{category}` slot runs through `friendlyCategoryInline()` in `copy.js` so raw keys like `foreign_policy` render as "foreign policy" — don't inline the raw value or you'll re-introduce the bug. Singleton groups (or `SAMPLE.alerts` rows missing `donation`/`vote` fields) render the original body unchanged — solo entries key on `id` so they never collide.

**Profile endpoint is the workhorse.** `/api/profile/{bioguide_id}` returns bio + funding + votes in one shot, fanning out ~5 upstream calls (Congress.gov bills + votes, OpenFEC totals + employers, WBMR). Cached in `ai_cache` under `profile:{bioguide_id}` for **6h** so repeat clicks on the same rep are instant; 6h is tight enough that recent votes and sponsored bills surface within a browsing session but loose enough that cold-load costs (2–5s) don't recur on every click. Dashboard uses lazy loading: `/api/reps/by-state/{state}` returns reps with `funding: null`, frontend fans out per-rep `/api/reps/{id}/funding-lite` calls — that endpoint is also `ai_cache`-backed under `funding_lite:{bioguide_id}` for **24h** on hits (campaign filings update quarterly so 24h is well inside freshness) but only **15 min** (`ttl_hours=0.25`) on the `has_data: False` negative sentinel, so transient WBMR/FEC misses self-heal on the next dashboard load instead of pinning "—" on a rep's card for a day. `openfec.py` has no in-process cache, so the persistent layer is the only thing standing between the OpenFEC fallback path and a fresh upstream call per page load.
- **Thin-WBMR cache guard.** `get_full_profile` will *not* write a cache row when WBMR returns a record but `top_industries == []` and `total_raised > $100K`. Real cause for the guard: Murphy got cached with `total_raised=$12.5M, top_industries=[], pac_total=$0`; the live call returned `$27.8M / 10 industries / $62K PAC`. WBMR served a stripped record for ~minutes and the 6h TTL pinned it. Heuristic catches the same signature next time without manual cache busts.
- **FEC ID gotcha:** reps with prior House service have their old House FEC ID first in `fec_ids[]` (e.g. Murphy `H6CT05124`, last cycle 2012); a naive `fec_ids[0]` pick returns empty totals under the current-cycle filter. The probe-all-IDs pattern (parallel `asyncio.gather` over every ID, pick first with non-empty `total_receipts`, fetch employers/contributors for the winning ID) is applied at every site that calls `get_candidate_totals` — `get_full_profile`, `get_rep_funding_lite`, `get_rep_detail`, `get_funding_detail`, and `scripts/demo.py`. Funding-lite's 15-min `has_data: False` TTL stays as WBMR-flake protection; the FEC half of the original motivation is now closed. Covered by `test_funding_lite.py` (stale-first/live-second + all-empty fall-through + WBMR short-circuit).
- **Principal committee gotcha:** FEC removed `principal_committees` from the `/candidate/{id}/` payload. `_get_principal_committee_id` now queries `/candidate/{id}/committees/?designation=P`; cycle filter omitted because the principal committee spans cycles and we always want the active one.
- **Employer noise filter:** `get_top_employers` skips `RETIRED / SELF / SELF EMPLOYED / SELF-EMPLOYED / HOMEMAKER / UNEMPLOYED / NOT EMPLOYED / NONE / NONE LISTED / N/A / INFORMATION REQUESTED / REQUESTED INFORMATION` — without this, individual-donor filer noise dominates the sort and the chatbot ends up citing "Self Employed" as a top funder.

**Events use a two-phase fetch.** `/api/events` fetches the Congress.gov committee-meeting list (URL stubs only), then `asyncio.gather`s up to 10 detail URLs in parallel. Title/date/location only come from detail. 5-min in-process cache. Event detail lazily fetches news (`/article` — NewsAPI primary, Guardian fallback) and AI summary (`/summary` — gpt-4o-mini, 2-sentence brief). `news.py`/`guardian.py` strip congressional boilerplate to build focused queries. After normalization, `fetch_events` drops past meetings (the header sells them as "upcoming") and sorts soonest-first; entries with an unparseable date stay (no way to confidently exclude). `_normalize_date` returns `(display, parsed_dt)` so this filter can run without re-parsing.

**AI features (stances, promises) require `OPENAI_API_KEY`.** Stances pull recent votes + sponsored bills, GPT identifies 4–6 policy areas with CONSISTENT/INCONSISTENT/MIXED/PENDING scores. Promises scrape the rep's `.gov` site (homepage + `/issues`, `/priorities`, `/about`, parallel `asyncio.gather`, 14k char cap), then GPT extracts stated positions and labels each KEPT/BROKEN/PARTIAL/UNCLEAR. Substantive-vote filtering shared via `is_substantive_vote()` in `congress_gov.py`.

**AI civics helper is a context-aware chat assistant — branded "Mamu."** `POST /api/assistant/chat` (auth-required) takes `{messages, context}` and runs gpt-4o-mini via `backend/api/assistant_chat.py`. Frontend exposes it as a dedicated **Mamu tab** in the bottom NavBar (sparkle icon, replaced the Search tab — Search now lives as a top-right magnifying-glass icon button in the Dashboard greeting row). The tab renders the full-screen `AssistantScreen` component in `App.jsx`. A floating "Mamu" pill also persists on every signed-in screen (hidden on Splash/Login/Create Account/Issue Select/Assistant itself); tapping the pill **navigates to the Mamu tab** rather than opening a bottom sheet, and the previous `AssistantSheet` modal is gone. Chat state (messages/input/sending/error) is lifted to App so tab navigation doesn't wipe the conversation; cleared on sign-out and via a Clear link in the screen header. Context source uses a `lastNonAssistantScreen` tracker (updated on every screen change that isn't ASSISTANT) so the chatbot still resolves rep/event/learn-to-vote context once the user is *on* the Mamu tab — without it the context would always read `{screen: "assistant"}`. Context inheritance: federal profile cluster (POLITICIAN_PROFILE + Funding/VotingHistory/PromiseScoring/Timeline/TakeAction) carries `{rep_id, rep_name}`; state profile cluster (STATE_REP_PROFILE + Voting/Stances/Promises/Alerts) carries `{state_rep_id}`; Event Detail carries `{event_title}`; Learn-to-Vote carries `{learn_to_vote_state}`; everything else falls back to `{screen}` only. `_build_context_block` reads cached data only — `profile:{bioguide_id}` for federal, `legiscan:profile:{pid}` for state, `scheduled_votes` row for bills — so chat round-trips stay <2s and never trigger live upstream fetches. Federal rep context always surfaces three complementary signals: industry/employer list, dollar totals (with explicit `$0` for zero sub-totals so the LLM doesn't compute residue), and the last two votes. Multi-turn memory means the full message list ships every send (capped at 30 turns; 400 on overflow). Responses cached 24h under `chat:{user_id}:{sha1(json.dumps(messages,sort_keys=True) + json.dumps(context or {},sort_keys=True))}`. `SYSTEM_PROMPT` scopes the bot to civics tutor and explicitly tells it to refuse off-topic asks and never invent specific facts not in the context block — empty cache fields render as "none cached" / "no recent votes cached" and the bot honors that. **Copy hook:** `COPY.assistant` in `copy.js` owns `pillLabel: "Mamu"`, `title: "Mamu"`, `subtitle: "Your civics-savvy Mamu — ask anything"`, plus the context-chip dict; rename there if the brand changes.

**Promise scraping uses a static fallback ladder.** `_scrape_with_fallbacks` in `promises.py` walks: (1) primary site (homepage + issue paths), (2) JSON-LD + `<noscript>` extraction from the primary HTML (recovers SPA shells that ship SEO text), (3) Wikipedia (`en.wikipedia.org/wiki/{name}`) with a 0.5s polite pause, (4) Ballotpedia (`ballotpedia.org/{name}`). Stops at the first rung yielding ≥400 chars; `attempted` list returned for diagnostics. State reps already use Ballotpedia as primary via `state_sites.derive_website`, so rung 4 is skipped when it would refetch the same URL. **No Playwright** — explicitly out of scope to keep deploy footprint small. Returns `{scraped: false}` only when every rung fails. Result includes `source_rung` so the frontend can attribute the source.

**Unified search.** `/api/search/unified?q=...&state=CT` fans out to federal (GitHub roster) and state (cached Legiscan roster) in parallel; results tagged `level: "federal" | "state"` so the frontend routes to the right profile screen.

**State legislators come from Legiscan.** Two-step roster fetch (`getSessionList` → newest → `getSessionPeople`), profile via `getPerson` + `getSponsoredList`. State deep-dive endpoints (`/votes`, `/stances`, `/promises`) mirror federal; promises uses Ballotpedia URLs from `state_sites.derive_website()`.
- **Legiscan gotcha:** `getSponsoredList` returns flat `sponsoredbills.bills` with `session_id` per item; `sponsoredbills.sessions` is metadata, not nested bills. Sort by `session_id` desc.
- **Rate-limit gotcha:** `_call` serializes through a process-wide lock with 1.05s spacing so `asyncio.gather` fan-outs stay under 1 req/sec free-tier limit. Cold-cache votes fetches run ~16s; don't remove without a paid-tier flag.
- **Chamber gotcha:** Legiscan normalizes any lower-chamber `role` to `chamber: "House"` even in Assembly states (NY/NJ/CA), so CSV rows must use "House", not "Assembly".

**State vote coverage uses a per-state index, not per-rep fan-out.** `legiscan.build_state_vote_index(state)` walks the top ~120 most-recently-active session bills, fetches `getBill`+`getRollCall`, and writes a `{people_id (str) -> [vote rows]}` reverse index to `ai_cache` under `legiscan:vote_index:{state}` (24h). `get_legislator_votes` reads that index first; non-sponsors who actually voted on recent bills now show real votes instead of empty lists. Built once per refresh cycle via `refresh._build_state_vote_index`; failure is logged but doesn't block pipeline. Falls back to the original sponsored-only probe when the index is absent (cold cache, fresh deploy). Per-bill (`legiscan:bill:{id}`) and per-roll-call (`legiscan:rollcall:{id}`) entries are also cached 24h so subsequent rebuilds reuse most of the prior work.

**AI + Legiscan results cached in SQLite (`ai_cache.py`).** Keys are namespaced: `promises:{id}`, `stances:{id}`, `promises:state:{pid}`, `stances:state:{pid}`, `profile:{bioguide_id}` (6h, full federal profile composite), `funding_lite:{bioguide_id}` (24h, dashboard fan-out), `legiscan:people:{state}` (7d), `legiscan:profile:{pid}` (24h), `legiscan:votes:{pid}` (24h), `legiscan:vote_index:{state}` (24h), `legiscan:bill:{id}` (24h), `legiscan:rollcall:{id}` (24h), `cat:{sha1(title+desc)}` (30d, AI categorization), `chat:{user_id}:{sha1(messages+context)}` (24h, civics-helper replies). Table created lazily; surviving server restarts. **Cache-bust gotcha:** poisoned entries stick for the TTL — wipe rows from `ai_cache` directly when fixing parsers (e.g., `DELETE FROM ai_cache WHERE cache_key LIKE 'profile:%'` after a profile-shape change).

**Alert scoring formula.** `scoring.py`: `S = (T*V) * (αD + βR + γA) * (1 + δN)` where T=topic match, V=vote proximity, D=donation magnitude, R=recency, A=anomaly z-score, N=news salience. `S>0.3` → alert, `S>0.6` → urgent. All signals stored on the alert for explainability.

**Alerts schema is polymorphic.** `donations`/`alerts`/`industry_baselines` key on `(actor_type, actor_id)` — `'federal'` (bioguide) or `'state'` (Legiscan people_id). `scheduled_votes` keys `(jurisdiction, state_code, bill_number)` so a CT bill and an NJ bill with the same number don't collide. Migrations in `db._migrate()` handle legacy DBs in place (table-swap idiom for PK/UNIQUE changes, runs *before* `executescript(SCHEMA)`). The `/api/alerts` endpoint accepts both `actor_type`/`actor_id` and legacy `bioguide_id` (auto-mapped to federal); responses include both fields.

**External IDs in a join table.** `external_ids(actor_type, actor_id, source, external_id, confidence, matched_at)` maps actors to FTM/OpenSecrets/FEC IDs. Adding a new data source doesn't churn the schema.

**Alerts carry `created_at` + `updated_at`.** `updated_at` is bumped on each pipeline re-confirmation so re-confirmed alerts read as fresh. Router compares against `datetime.now(timezone.utc).replace(tzinfo=None)` because SQLite's `CURRENT_TIMESTAMP` is UTC-naive — using `datetime.now()` skews relative-time strings by the local-UTC offset.

**State donations come from FTM lifetime aggregates, not itemized rows.** `ingest_ftm.py` per-actor loop: cached Legiscan roster → resolve eid → fetch industry breakdown via `dataset=contributions&gro=d-cci&c-t-eid=<EID>` → translate via `catcode_map.industry_for_ftm_name` (drops `_ignore` buckets) → write one `donations` row per industry tagged `actor_type='state'`. `donation_date = today` because FTM grouped aggregates are lifetime-only (year filters return 0 records when combined with grouping). Keeps rows inside the 180-day lookback. Dedup key: `FTM:{eid}:lifetime:{industry_slug}`.
- **Connection gotcha:** holding `with connect()` across `await`s starves `ai_cache` writes and triggers SQLite "database is locked." Open per-actor.
- **EID lookup uses an offline directory.** Live `_live_find_eid` is a stub — FTM's candidates-dataset state filter is undocumented and none of `s-y-st`, `s`, `c-r-s`, `c-t-s`, `c-r-osj` honor it empirically. Workaround: `backend/data/ftm_eids.csv` (loaded by `ftm_directory.py`), curated `(state, chamber, name, eid)` rows consulted before live and before `SAMPLE_FTM_EIDS`. Fuzzy match (SequenceMatcher ≥ 0.78) with chamber as disambiguator. Adding mappings is manual: look up on followthemoney.org, copy eid from URL, append CSV row. `site:followthemoney.org "LASTNAME, FIRSTNAME"` is the most reliable Google query (FTM's title format). `scripts/find_ftm_eids.py CT House` walks the Legiscan-roster gap interactively — opens each Google search in the browser and prompts for the eid to append. Most incumbents in both chambers across CT/NY/NJ/CA/MA have entries; the persistent gaps are 2024-cycle freshmen FTM hasn't indexed yet plus a handful of common-name reps Google can't disambiguate. **Legiscan NJ quirk:** the NJ "House" roster mixes Senate members in with the Assembly — when picking gaps, tag rows with `SD-` districts as Senate, not House.
- **Cache pollution gotcha:** test runs that misconfigure `db.DB_PATH` will write `[]` sentinels to production `ai_cache`. If FTM matches stop returning, wipe rows where `cache_key LIKE 'ftm:%'`.
- **Quota gotcha:** FTM free tier returns HTTP 200 with `{"error": "...account has reached its free API call limit..."}`. `_ftm_get` raises `FTMUpstreamError`; `get_industry_aggregates` catches it, returns sample data for *just that call*, and does NOT cache. So a transient quota hiccup doesn't pin demo numbers under the live key for 24h.

**Upcoming-vote feeds use bill-status as a proxy** (no clean upstream calendar exists for either jurisdiction).
- State (`ingest_state_votes.py`): Legiscan masterlist filtered to `STATUS_ENGROSSED`. **Title-only categorization** via `state_categories.py` keyword regex (14 categories: environment, healthcare, economy, defense, infrastructure, technology, labor, agriculture, housing, education, immigration, firearms, elections, foreign_policy). Description fallback was removed — long policy summaries produced too many incidental matches. **AI fallback (`ai_categorize` in `state_categories.py`) runs only on regex misses** — gpt-4o-mini, title+description, single-word output constrained to the 14 categories or "none", cached 30d under `cat:{sha1(title+desc)}`. Result tracked in `stats["ai_recovered"]`; uncategorized titles still logged so the regex can grow over time. Behavior identical to pre-AI when `OPENAI_API_KEY` is unset. No per-bill `getBill` (would burn the 30k/mo quota). Masterlist cached 6h.
- Federal (`ingest_federal_votes.py`): `/v3/bill/{congress}` sorted `updateDate+desc`, paginated 4×250, filtered to `latestAction.text` matching `_FLOOR_IMMINENT_PATTERN` in `congress_gov.py`. Resolution-type bills (`hres`/`sres`/etc.) dropped as symbolic. Reuses the same keyword categorizer — categories are jurisdiction-agnostic. `foreign_policy` was added as the 14th category specifically to recover the federal residue (sanctions/treaty/country-name bills); positioned LAST in `CATEGORY_KEYWORDS` so existing China-economy / NATO-defense / Uyghur-labor routing isn't disturbed. Acronym-only titles (SPUR/SERV/etc.) still need the AI fallback. No caching — meant to run on cron, not ad hoc.
- Both: `scheduled_date = status_date + DEFAULT_VOTE_LEAD_DAYS` (14d). Stalled bills with past projected dates get bumped to today so V doesn't go to zero.
- **Stale-row purge** at end of each ingest: deletes `scheduled_votes` rows (and their downstream `alerts`) whose `bill_number` isn't in the current categorized set. Gated on `keepers` being non-empty so a quota error can't nuke the table. Federal ingest *replaces* `seed.py` — re-seeding after an ingest will get re-purged on the next run.

**Pipeline runs federal and state independently.** `pipeline.py` calls `_run_for_jurisdiction` once per actor_type so a federal rep's donations never pair with state bills. Baselines computed in one pass with `GROUP BY actor_type`.

**Stale alert sweep.** After scoring, `_sweep_stale_alerts` deletes any non-dismissed alert whose `(donation_id, vote_id)` pair wasn't refreshed this run. Covers donation aged out, vote already happened, or score recalibrated below threshold. **Dismissed alerts are kept** so user-suppressed history doesn't resurface. Per-`actor_type` scope.

**State alert calibration.** Without it, every state alert lands above urgent because (1) FTM donations are lifetime-stamped today so R saturates to 1.0; (2) state donation pools are too sparse to clear `BASELINE_MIN_SAMPLES=3`, so A defaults to 0.5. Pipeline passes state-only `proxy_donation_r=0.4` (career-flat) and `no_baseline_a=0.0` (honest "no signal") into `score_alert`. Federal is unchanged. Env-overridable: `ALERTS_PROXY_DONATION_R`, `ALERTS_NO_BASELINE_A_HONEST`.

**Auth is local-only — bcrypt + opaque session tokens.** `/api/auth/{signup,login,logout,me}`; 32-byte tokens, 30-day TTL in `sessions` table, `Authorization: Bearer` from frontend. `users` holds `email`, `password_hash`, `name`, `state`, `issues` (JSON list of category keys — see above). Profile updates via `PATCH /me`. Schema lazy-init via `_ensure_schema()`. `get_current_user` is the strict dependency (raises 401); `get_current_user_optional` is its sibling that swallows the 401 and returns `None` — use it for endpoints that personalize when signed in but still serve anonymous traffic (the only consumer today is `/api/upcoming-votes`). Hardening: per-IP/email throttle (8 fails / 15min, 429), constant-time login (bcrypt vs `_DUMMY_PASSWORD_HASH` for unknown emails), `DELETE /me` requires re-auth password, signup rejects ~30 trivial passwords, opportunistic session prune. Missing: email verification, password reset (need SMTP infrastructure).

**Guest mode is a temporary demo affordance.** Login screen has a "Continue as guest" link below the create-account row. Tapping it calls `handleEnterGuest` in App, which sets `currentUser = { name: "Guest", state: "CT", issues: ["healthcare", "environment"], is_guest: true }` (no token written) and navigates to Dashboard. Auth-touching handlers check `currentUser?.is_guest` and bypass the API: `handleSaveState` / `handleSaveIssues` update local React state only; `handleSignOut` skips `api.logout`; `handleDeleteAccount` degenerates to `handleSignOut`. Mamu tab swaps the chat body for a sign-up CTA card since `/api/assistant/chat` is auth-required (`AssistantScreen` returns early on `isGuest`). Settings's Account block shows a "Browsing as guest" card with Sign Up / Log In + Exit guest mode buttons in place of the signed-in card; Delete Account is hidden. **Marked temporary by the user** — rollback is a grep-and-rip of `is_guest` / `isGuest` / `handleEnterGuest` / `onEnterGuest`.

## Conventions

- Backend: `async`/`await` everywhere, `httpx.AsyncClient`, exponential backoff on 429s.
- New external APIs go in `backend/api/` as their own module, composed in `server.py`.
- Frontend stays single-file. Don't add React Router.

## Deploy / PWA

**Frontend:** Vercel + PWA. `manifest.webmanifest` + `sw.js` + `<link rel="manifest">` make it installable on Android Chrome and iOS Safari "Add to Home Screen." SW registers only when `hostname !== "localhost"` (dev iteration isn't fighting stale caches); pre-caches app shell, network-first otherwise, **`/api/*` deliberately bypassed** so live data never caches. `vercel.json` pins SPA rewrites + `Cache-Control: no-cache` on `/sw.js`.

**PWA-mode UI is a different render path.** Module-level `_IS_PWA_AT_BOOT` snapshot of `display-mode: standalone` + iOS legacy `navigator.standalone`. `App` branches on it: PWA mode returns `renderScreen()` directly with **no dev chrome** (no "Live Prototype" title, screen pills, or phone-frame container). `s.phone` branches at module load: browser is 393×852 framed; PWA is `100% / 100%`. `StatusBar` and Settings "Backend Status" return null in PWA. Don't use a runtime hook — display-mode doesn't change at runtime and `s` needs to branch synchronously.

**Three iOS PWA gotchas in `index.html` standalone CSS** (gated by `@media (display-mode: standalone)`):
1. **Body-lock** to kill rubber-band: `html, body { height: 100%; overflow: hidden; overscroll-behavior: none }` + `body { position: fixed; inset: 0 }`. Scroll happens inside `#root { overflow-y: auto; -webkit-overflow-scrolling: touch; overscroll-behavior: contain }`.
2. **`s.phone` must be `100%`, not `100vw/100vh`** — body has `padding: env(safe-area-inset-*)` already shrinking to safe area; vw/vh reach back into dynamic-island/home-indicator zones and squish proportions.
3. **`status-bar-style: default` does NOT reserve space** on modern iOS — page extends behind the status bar. `padding-top: env(safe-area-inset-top)` is still required. Body bg in PWA mode is `#fdfbf7` (matches `colors.bg`) so safe-area regions read continuous instead of black bands.

**Backend deploy:** `Procfile` works on Railway/Render/Heroku/Fly. Required for full features: `DATA_GOV_API_KEY`, `OPENAI_API_KEY`, `NEWSAPI_KEY`, `GUARDIAN_API_KEY`, `LEGISCAN_API_KEY`, `FTM_API_KEY`. Frontend needs `VITE_API_BASE=https://your-backend-url` in Vercel env before deploying. CORS is `allow_origins=["*"]` — fine for read-only, tighten if surface changes.

- **Env-var split gotcha:** the backend API keys live on Railway (or wherever the FastAPI app runs); `VITE_API_BASE` lives on **Vercel** because Vite inlines `import.meta.env.*` at build time, not runtime. Adding it to Railway does nothing. Symptom of a missing `VITE_API_BASE` on iOS PWA: login POST fails with "Load failed" — fetch goes to relative `/api/auth/login`, Vercel's SPA rewrite returns `index.html`, and the JSON parse failure presents as a network-style error. Fix is dashboard-only (Vercel → Settings → Environment Variables → Production) + redeploy + delete/reinstall the PWA so the SW picks up the new bundle.

- **Persistence caveat:** SQLite alerts DB lives at `backend/data/whoboughtmyrep.sqlite`. Ephemeral filesystems (Vercel functions, Render free tier) lose data on redeploy — use Railway/Fly volumes/Render paid disk.
- **Cron caveat:** Railway/Render have native scheduled jobs; Fly uses scheduled machines. `refresh.py` is the entrypoint.

## Gotchas

- `python -m backend.alerts.*` must run from **project root**, not inside `backend/`. Imports rely on it.
- `/api/alerts/*` 503s until `python -m backend.db` runs and the pipeline has populated.
- Vite proxies `/api/*` → `:8000`. Don't prefix `http://localhost:8000` in frontend fetches.
- For prod frontend builds, `VITE_API_BASE=https://backend-url` before `npm run build`.
- OpenFEC `candidate → principal committee` is fuzzy — see `get_principal_committee` fallback chain in `ingest_fec.py` (scoped cycle, then unscoped, then committee lookup by candidate_id).
- State alerts need cached Legiscan rosters to attribute donations. After a cache wipe: hit the state-rep screen or run `ingest_state_votes` to repopulate before `pipeline.py`.
